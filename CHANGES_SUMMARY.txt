POLLING: explicación detallada (cómo funciona en el código y por qué)

Objetivo del polling implementado:
- Mantener la UI del chat actualizada en (casi) tiempo real consultando al servidor periódicamente.

Dónde está implementado:
- Cliente: `Views/Conversaciones/Details.cshtml` (funciones JavaScript `fetchMessages`, `renderMessages`, `sendMessage`).
- Servidor: `MensajesController.ByConversacion` (GET) y `MensajesController.CreateMessage` (POST).

Comportamiento paso a paso (cliente):
1) Inicialización
   - Al cargar la vista de detalles, se ejecuta `fetchMessages()` una primera vez para traer el historial.

2) `fetchMessages()` (polling controlado)
   - Llama `fetch('/Mensajes/ByConversacion?conversacionId=' + conversacionId, { credentials: 'include' })`.
   - Usa `credentials: 'include'` para enviar cookies y mantener la sesión (autenticación por cookies).
   - La petición está dentro de un `try/catch/finally`.
   - En `finally` se programa la siguiente llamada con `setTimeout(fetchMessages, 1000)` (1 segundo) — esto garantiza que no haya solapamiento: la siguiente petición se programa solo cuando la actual finaliza (éxito o fallo).
   - Si hay error, se captura en `catch` y se reintenta tras 1s (backoff simple no implementado; se podría añadir).

3) `renderMessages(messages)`
   - Reemplaza el contenido del `chatWindow` con la lista completa de mensajes recibidos (por simplicidad).
   - Para cada mensaje se crea un nodo DOM mostrando remitente, contenido, hora y el `estado` (Enviado/Entregado/Leído) devuelto por el servidor.
   - Al final se hace `scrollTop = scrollHeight` para mantener vista en el final del chat.

4) `sendMessage()`
   - Envía POST a `/Mensajes/CreateMessage` con JSON { Contenido_Mensaje, ConversacionId, RemitenteId } y `credentials: 'include'`.
   - Si el POST responde `success`, limpia el input y llama `fetchMessages()` para recuperar mensajes actualizados inmediatamente (optimistic refresh).
   - El servidor también crea registros `MensajeLectura` para marcar estados básicos.

Por qué implementamos setTimeout en `finally` en vez de `setInterval`:
- `setInterval` genera llamadas a intervalos fijos sin respetar si la petición anterior terminó; en redes lentas pueden solaparse peticiones (doble carga, respuestas fuera de orden).
- Con `setTimeout` al final de la petición nos aseguramos que la siguiente petición sólo ocurra una vez que la anterior terminó (control de concurrencia simple).

Limitaciones y mejoras recomendadas:
- Actualmente el cliente solicita todo el historial con cada poll. Escalar mejor:
  - Implementar `GET /ByConversacion?afterId={lastMessageId}` para devolver solo mensajes nuevos. Mantener `lastMessageId` en el cliente y actualizarlo con el último ID recibido.
  - O usar `If-Modified-Since`/ETag para reducir payload cuando no hay cambios.
- Backoff en errores: si el servidor falla repetidamente, aumentar el intervalo (2s, 5s...) hasta recuperar.
- Detección de visibilidad: pausar polling cuando la pestaña está en background (usar Page Visibility API) para reducir consumo.
- Mejor alternativa: `SignalR` (WebSockets) para real-time verdadero, menos latencia y eficiencia.
- Considerar deduplicación: si renderizas todo el historial, usa `lastMessageId` para añadir solo mensajes nuevos y evitar reconstruir todo el DOM.
- Seguridad: `credentials: 'include'` requiere cookies de autenticación; si usas tokens bearer, ajusta cabeceras.
